# Learning Roadmap

## Purpose

The learning roadmap turns the handbook into a structured education program. It defines how learners move from orientation to portfolio-ready projects without requiring every audience to study every chapter in the same order.

## Learning Model

The handbook uses a progressive learning model:

1. Orientation: understand the handbook, tools, standards, and expected evidence.
2. Build: complete a small working product before deep theory.
3. Develop: improve engineering practices and code quality.
4. Validate: add testing, quality gates, and objective evidence.
5. Secure: introduce threat thinking, secure configuration, and review habits.
6. Augment: use AI responsibly for design, development, validation, and documentation.
7. Operate: prepare software for production reliability and maintainability.
8. Scale: connect the work to enterprise DevSecOps and architecture practices.
9. Demonstrate: complete capstone evidence and explain decisions.

## Learning Tracks

The roadmap supports both sequential learners and role-based learners.

- Sequential learners complete the entire handbook from Getting Started through Phase 06.
- Role-based learners follow a curated path based on career goals.
- Instructors may combine phases into semester modules or bootcamp sprints.
- Startup teams may use only the sections that support current product maturity.

## Evidence-Based Completion

Progress is based on evidence, not passive reading.

Acceptable evidence includes:

- Completed chapter checklist.
- Completed lab checklist.
- Quiz score.
- Screenshot submission.
- Project milestone.
- Reflection answer.
- Instructor review note.

## Recommended Completion Milestones

| Milestone | Completion target |
| --- | --- |
| Foundation | Getting Started and Phase 00 |
| Engineering Core | Phases 01 and 02 |
| Secure Delivery | Phase 03 and selected Phase 06 topics |
| AI-Augmented Delivery | Phase 04 and prompt documentation evidence |
| Production Readiness | Phase 05 and operations validation |
| Portfolio Readiness | At least one capstone project |

## Roadmap Governance

Do not change the learning roadmap casually. Changes affect navigation, study plans, instructor pacing, certification mapping, and capstone expectations.

Roadmap changes require:

- Editorial review.
- Learning experience review.
- Navigation review.
- Versioning note.
